## 오케스트레이션 흐름

```
[1] 요구사항 파악
        ↓
[2] code-researcher 에이전트 → 기존 코드 분석
        ↓
[3] 구현 계획 수립 → 사용자 확인
        ↓
[4] 필요시 domain-scaffolder 에이전트 → 스캐폴딩
        ↓
[5] 비즈니스 로직 구현 (메인 스레드)
        ↓
[6] 필요시 db-migrator 에이전트 → 스키마 변경
        ↓
[7] 웹 UI 변경 (필요시)
        ↓
[8] /git-flow → 커밋
```

## 단계별 실행 절차

### [1] 요구사항 파악
- 어떤 기능인지, API/Web/DB 중 어디에 영향 주는지 확인
- 모호하면 사용자에게 질문

### [2] 코드 분석 (서브에이전트)
`code-researcher` 에이전트에 위임:
- 관련 기존 도메인/컴포넌트 파악
- 따라야 할 패턴 확인
- 결과를 메인 스레드에 요약 보고

### [3] 구현 계획
분석 결과를 바탕으로 계획 수립:
- 새 도메인 필요 여부
- DB 스키마 변경 여부
- API 엔드포인트 목록
- 웹 UI 변경 범위
→ **사용자에게 계획 확인 후 진행**

### [4] 도메인 스캐폴딩 (서브에이전트, 필요시)
새 도메인이 필요하면 `domain-scaffolder` 에이전트에 위임:
- 도메인명과 설명 전달
- 생성 완료 보고 받기

### [5] 비즈니스 로직 구현
메인 스레드에서 직접 구현:
- Service 로직
- Repository 쿼리
- Controller 엔드포인트

### [6] DB 마이그레이션 (서브에이전트, 필요시)
스키마 변경이 필요하면 `db-migrator` 에이전트에 위임:
- 변경 내용 전달
- 스키마 수정 완료 후 사용자가 migrate 명령 실행

### [7] 웹 UI
API 변경에 맞춰 `web/src/` 업데이트

### [8] 커밋
`/git-flow` 스킬 실행

## 원칙

- 각 단계 완료 후 사용자에게 진행 상황 보고
- 불확실한 설계 결정은 구현 전에 반드시 확인
- 단계 실패 시 다음 단계로 넘어가지 않고 문제 해결 후 재시도
